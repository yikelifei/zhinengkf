# 新版桌面客服平台

这是新版 Windows 桌面客服平台工程，承接这条业务闭环：

微信客户提出礼盒设计需求 -> 客服平台识别预算和场景 -> SKU 商品库自动搭配礼盒 -> 调用设计平台出礼盒效果图 -> 客服审核/发送 -> 客户选图 -> 生成报价草稿。

## 目录说明

- `apps/api`: NestJS 本地 API，负责商品库、礼盒搭配、设计任务、设计平台回调、报价草稿和发送队列。
- `apps/web`: Next.js 客服工作台界面。
- `apps/electron`: Electron 桌面壳。
- `packages/rules`: 可单独测试的业务规则，包含预算解析、高价值判断、SKU 搭配、设计任务状态和客户选图识别。
- `prisma/schema.prisma`: 数据模型，覆盖微信账号、客户、会话、SKU、设计任务、候选图、报价草稿和发送队列。
- `tools`: 本地端口启动、停止和模拟设计平台工具。
- `.runtime`: 本地运行时记录和日志，已加入 `.gitignore`，不需要提交。

## 推荐启动方式

从项目根目录双击：

```bat
run_desktop.bat
```

启动器会先检查 Node.js、npm、端口占用和日志目录。第一次运行时，如果 `desktop/node_modules` 不存在，会自动安装依赖。

启动后访问：

- 客服工作台：`http://127.0.0.1:3100/`
- API 健康检查：`http://127.0.0.1:3200/api/health`
- 模拟设计平台：`http://127.0.0.1:3700/v1/health`

停止端口：

```bat
stop_desktop.bat
```

查看状态：

```bat
cd desktop
npm.cmd run ports:status
```

## 数据模式

默认使用本地 JSON 数据模式：

- `USE_LOCAL_STORE=true`
- 数据文件：`desktop/.runtime/local-store.json`
- 第一次启动 API 时自动生成演示微信账号、客户、会话和 SKU。
- 适合小白本机试用、演示业务流程、开发前端交互。

重置演示数据：

```bat
npm.cmd run data:reset
```

后续正式部署再切到数据库模式：

- PostgreSQL
- Redis
- `desktop/.env` 里的 `DATABASE_URL`
- `USE_LOCAL_STORE=false`

第一版 Prisma schema 固定 PostgreSQL，字段设计已尽量避免 PostgreSQL 独占能力，后续可以为 MySQL 做适配。

## 聊天训练演示

页面底部有 `Agent 中心` 和 `训练中心`：

1. 在训练中心粘贴聊天记录。
2. 格式建议使用 `客户：...` 和 `客服：...` 一问一答。
3. 点击 `导入训练`。
4. 系统会自动提取客户问题、客服回答、场景分类、适配 Agent 和 Skill 提示。
5. 查看 `Skill 进化预览`，系统会从 70 分以上的高质量样本中提炼可复用 Skill 建议。
6. 如果同一 Agent 已有同名 Skill，系统会更新版本；没有同名 Skill 时才会新增，避免重复堆积。
7. 默认只勾选样本数不少于 2 条且置信度 80 分以上的建议；样本少或置信度低的建议会保留展示，但需要人工主动勾选。
8. 点击 `应用已选 Skill` 后，样本、知识条目和 Skill 会进入本地 JSON 数据，刷新后仍然保留。
9. 后端接口也有安全闸门：不传 `includeNeedsReview=true` 时，只会真正应用高可信建议，低样本或低置信度建议会被返回为需复核，不直接写入 Skill。
10. 场景不明确时生成的确认话术不会被提炼成“物流安抚、售后方案、设计需求确认”等业务 Skill，只会沉淀为 `防乱回复`；对应知识条目也不会参与普通回复参考，避免泛化话术污染客服回答。
11. 训练样本会返回质量标签：`可训练`、`待人工复核`、`低分需复核`、`防乱回复样本`、`已禁用`。前端会直接显示原因，方便人工判断样本能不能进入业务 Skill。

训练中心会显示一层训练概览，用来判断智能体是否真的在进化：

- `纠错样本`：客服手动纠正场景后生成的 `route_correction` 样本。
- `可生成 Skill`：当前高分样本可提炼出的 Skill 建议数量。
- `待复核`：低质量或不确定样本，需要人工确认后再进入训练。
- `智能体覆盖`：每个 Agent 有多少训练样本、多少纠错样本、多少 Skill 建议。
- `Skill 进化预览`：在真正应用 Skill 前，先看到每条建议是新增还是更新、来自多少样本、适用哪些场景、参考了哪段客户/客服对话；可按 Agent 筛选，系统默认只勾选高可信建议，低样本或低置信度建议需要人工复核后再勾选应用。

每条训练样本支持人工复核：

- `确认训练`：状态改为 `ready`，允许参与 Skill 生成和知识匹配。
- `退回复核`：状态改为 `review`，暂时不参与 Skill 生成和知识匹配。
- `禁用`：状态改为 `rejected`，不参与 Skill 生成和知识匹配，但保留原始记录和审核日志。
- `编辑样本`：可以修正 Agent、场景、客户问题、标准回复、评分和 Skill 提示，再保存为待复核或直接确认训练。

训练相关接口：

- `GET /api/training/chat-imports`：查看导入批次。
- `POST /api/training/chat-imports`：导入聊天记录。
- `GET /api/training/samples`：查看训练样本。
- `GET /api/training/overview`：查看训练概览、纠错样本统计和 Agent 覆盖情况。
- `POST /api/training/samples/:id/review`：复核训练样本，确认训练、退回复核或禁用。
  - 同一个接口也支持带上修正字段：`agentKey`、`scene`、`customerText`、`idealReply`、`score`、`skillHints`。
- `GET /api/training/skill-suggestions`：预览可提炼的 Skill。
- `POST /api/training/skill-suggestions/apply`：把建议应用到 Agent Skill；可传 `suggestionKeys` 只应用人工勾选的建议。默认只应用高可信建议；如果人工复核后确实要应用低样本或低置信度建议，需要同时传 `includeNeedsReview=true`。

当前默认内置这些 Agent：

- 售前转化 Agent
- 礼盒设计 Agent
- 物流异常 Agent
- 售后安抚 Agent
- 尺码推荐 Agent
- 未分类兜底 Agent

## 多账号发送安全演示

页面底部有 `多账号控制` 和 `发送安全队列`：

1. 点击 `创建演示发送`，系统会创建一条待发送任务。
2. 点击 `错误窗口校验`，系统会模拟当前微信窗口停在另一个客户会话，任务会被拦截。
3. 点击 `正确窗口校验`，系统会模拟账号、聊天对象、最近客户消息都匹配。
4. 点击 `通过后发送`，只有校验通过的任务才会标记为已发送。

当前校验规则：

- 微信账号必须匹配。
- 当前聊天对象必须匹配。
- 最近消息或客户 ID 必须匹配。
- 单个微信账号内必须按队列顺序串行发送。

发送任务进入队列前还会先做绑定校验：

- `wechatAccountId` 必须等于会话所属微信账号。
- `conversationId` 必须是真实存在的会话。
- 如果任务关联设计任务，设计任务必须属于同一个会话、同一个客户、同一个微信账号。
- 如果任务关联报价草稿，报价草稿必须属于同一个客户，并且报价绑定的设计任务不能和发送任务冲突。

这层校验发生在任务创建时，错误任务不会进入发送队列；发送前仍然会继续做窗口快照和最近消息校验。

第一版不会绕过平台规则去登录、多开或真实控制微信发送；这里先把商业落地最关键的“不要发错账号、不要发错客户”的安全闸门做成可验证逻辑。

## 智能体路由决策演示

页面底部有 `路由决策中心`：

1. 输入客户最新一句话。
2. 点击 `判断谁来处理`。
3. 系统会识别场景、预算、客户价值、缺失信息和风险词。
4. 系统会给出处理方式：智能体处理、先补信息、转人工。
5. 系统会给出一条不生硬的建议回复。

如果场景信号弱或同一句话同时像多个场景，系统会先追问客户重点；同一会话下一句如果回复“先处理退款”“我是问发票”“第二个”等内容，会自动沿用最新一条仍待确认的上下文，把场景修正为清晰场景后再交给对应 Agent。客户已经澄清后，旧追问不会继续影响后面的新问题。

如果客服发现系统判错场景，可以在路由结果里直接点“改为某个 Agent”。系统会更新这条路由的场景、保留原判定记录，并自动生成一条 `route_correction` 训练样本和知识条目，后续可继续参与 Skill 生成和回复参考。

这些纠错样本会进入训练中心概览，系统会统计哪些 Agent 经常被纠正、哪些场景最容易误判、当前能生成多少 Skill。这样不是简单套模板，而是把客服确认过的判断沉淀成下一轮路由和回复的依据。

示例：

- `端午员工福利礼盒，每盒180元，做50份，想看真实摆拍效果图，logo已发` -> 礼盒设计 Agent，智能体处理。
- `我想做礼盒效果图` -> 礼盒设计 Agent，先补预算、数量、用途等信息。
- `企业伴手礼总预算1万，做100份` -> 高价值，转人工审核。
- `我要投诉，东西破损还不给赔偿` -> 售后安抚 Agent，风险场景转人工。

路由评估现在会同时读取当前 Agent 的 Skill 和训练知识样本：

- `appliedSkills`：本次回复草稿用到了哪些 Skill。
- `knowledgeMatches`：匹配到的历史训练样本，给客服看参考依据。
- `replyDraft`：回复来源、下一步动作和安全检查。
- `suggestedReply`：增强后的客户可读回复草稿。信息不完整时只追问，不会直接调用设计或报价；高价值和风险场景仍然转人工。

## 入站客户消息自动处理

新增接口：

```http
POST /api/wechat/inbound/messages
```

用途：模拟真实微信收到一条客户消息后的完整处理链路。

请求示例：

```json
{
  "conversationId": "conversation_demo_1",
  "text": "端午员工福利礼盒每盒180元，做50份，想看真实摆拍效果图，logo已发",
  "assetIds": ["asset_demo_logo"]
}
```

处理顺序：

1. 写入本地消息记录，并更新会话最后消息时间。
2. 调用路由评估，判断 Agent、场景、预算、是否高价值、缺失字段。
3. 读取当前 Agent 的 Skill 和训练样本，生成增强回复草稿。
4. 生成自动化计划：
   - 高价值或风险问题：锁定人工处理并发提醒。
   - 信息不完整：排队一条追问回复。
   - 礼盒设计但没有真实素材：只追问素材，不创建出图任务。
   - 礼盒设计且有真实素材：自动推荐 SKU 搭配，创建设计草稿，并排队客户回复。
   - 其他普通场景：排队智能体回复草稿。

注意：即使生成了回复，也只是进入微信安全发送队列，不会绕过账号、聊天对象和最近消息校验。

## 设计草稿自动提交

新增接口：

```http
POST /api/design-jobs/auto-submit-drafts
```

用途：扫描当前设计草稿，把已经满足条件的低风险任务自动提交到设计平台，继续复用原来的 `POST /api/design-jobs/:id/submit` 提交流程。

会自动提交的条件：

1. 任务状态必须是 `draft`。
2. 不是高价值客户任务。
3. 已经有预算、礼盒搭配、设计类型、场景或客户需求。
4. 已经绑定真实客户素材，不能只靠“客户说 logo 已发”这种文字。

会跳过的情况：

1. 高价值任务，继续留给人工审核。
2. 缺预算、缺搭配、缺需求或缺真实素材。
3. 任务已经提交、出图中、完成、失败或取消。

工作台提醒区有 `自动提交草稿` 按钮。点击后会显示本次检查了多少草稿、提交多少、跳过多少、失败多少。提交成功后仍然会走设计平台上传素材、创建任务、等待消息、轮询结果和后续快速确认流程，不会新增另一套接口。

## 高价值客户转人工

高价值默认线是 `10000 元`，只要订单总额或单份金额任一达到这个线，就不走低价值自动化。

会转人工的入口：

1. 入站客户消息识别为高价值时，不创建设计任务，不排队自动回复，只生成提醒并锁定人工处理。
2. 直接创建高价值设计任务时，任务状态会直接进入 `manual_review`。
3. 高价值客户选图后，不自动生成报价草稿，继续提醒人工确认报价、交期和跟进话术。
4. 历史遗留的高价值草稿可以用扫描接口批量转人工。

新增扫描接口：

```http
POST /api/design-jobs/scan-high-value-handoffs
```

工作台提醒区有 `高价值转人工` 按钮。它只处理高价值且仍处在可接管状态的任务；已经在人工审核、已经发送、已取消或已生成报价的任务不会重复处理。

转人工时系统会同时做三件事：把设计任务改为 `manual_review`，把对应会话标记为 `manualLocked=true`，并写入一条 `ReviewLog`。这样高价值客户进入人工接管后，后续自动回复和自动发送不会继续抢处理权。

会话人工接管也有独立接口：

```http
POST /api/wechat/conversations/:id/manual-lock
```

请求示例：

```json
{
  "locked": false,
  "reviewer": "人工客服",
  "reason": "manual_resolution_from_workbench",
  "note": "人工客服已处理完该会话，恢复自动化判断。"
}
```

`locked=true` 表示人工接管，`locked=false` 表示人工处理完毕后解除锁定。每次切换都会写入 `ReviewLog`，方便复盘是谁、为什么接管或释放了客户会话。

## 低价值自动处理

新增接口：

```http
POST /api/design-jobs/auto-process-low-value
```

用途：把低价值客户的设计任务继续往前推一段，减少人工点击。

目前会自动做两件事：

1. 调用 `auto-submit-drafts`，把资料齐全的低价值设计草稿提交到设计平台。
2. 扫描已经出图、状态为 `quick_confirm` 的低价值任务，把候选图加入微信安全发送队列。

不会自动做的事：

1. 不处理高价值客户，高价值任务继续人工审核。
2. 不绕过微信发送安全校验。
3. 不直接操作平台账号发消息，只是创建待发送任务。
4. 没有候选图、没有微信账号、没有会话绑定的任务会跳过。

工作台提醒区有 `低价值自动处理` 按钮。后续可以把这个接口接成定时巡检，让系统每隔一段时间自动跑一遍。

## 报价/订单草稿演示

页面底部有 `报价/订单草稿`：

1. 在设计任务里生成候选图。
2. 选择客户喜欢的图。
3. 低价值客户选图明确后，系统会自动生成报价草稿，并把设计任务标记为 `quote_created`。
4. 高价值客户选图后不会自动报价，会进入人工审核，避免大单报价误发。
5. 如果需要人工重新生成或补生成，也可以点击 `生成报价`。
6. 报价草稿会显示客户、场景、数量、单价、总价、成本利润和利润率。
7. 可以标记 `定金`、`已付成交` 或 `人工跟进`。

第一版这里不是完整财务订单系统，先承接“客户选图后生成报价/成交草稿”的业务闭环。后续可以继续扩展合同、发票、收款流水、生产单和发货单。

## 报价发送安全队列演示

报价不是直接改成“已发送”，而是先进入微信安全发送队列：

1. 普通报价可以在 `报价/订单草稿` 里点击 `发送报价`。
2. 高价值报价在 `人工审核中心` 点击 `通过并入队` 后，会自动创建一条报价发送任务。
3. 发送任务的 payload 类型是 `quote`，会绑定 `quoteDraftId`，避免报价和客户会话断链。
4. 发送前仍然校验微信账号、当前聊天对象、最近消息/客户 ID 和单账号队列顺序。
5. 点击 `执行干跑发送` 后，不会真实操作微信，只会记录一次发送尝试，并把任务标记为 `dry_run`。
6. `dry_run` 代表“安全校验和审计已跑过，但客户还没有收到消息”。如果要真实发送，需要重新排队后用 `windows_bridge` 执行。
7. 切换到 `windows_bridge` 后，任务会先进入 `sending`，必须等桥接回执校验通过后才会变成 `sent`，报价草稿才会从 `send_queued` 回写为 `sent`。

这一块后续可以把 `dry_run` 适配器替换成真实 Windows 微信操作适配器，但业务层接口、队列、安全闸门和审计记录不用重做。

## 微信发送适配器

发送执行层现在已经拆成适配器：

- 默认适配器：`dry_run`
- 当前配置：`WECHAT_SEND_ADAPTER=dry_run`
- 查看当前适配器：`GET /api/wechat/send-adapter`
- 执行当前适配器：`POST /api/wechat/send-tasks/:id/execute`
- 强制干跑执行：`POST /api/wechat/send-tasks/:id/execute-dry-run`

`dry_run` 只做窗口校验、队列顺序校验和审计记录，不会真实操作微信，也不会把任务当成真实已发送。

旧的直通标记接口 `POST /api/wechat/send-tasks/:id/mark-sent` 和 `POST /api/wechat/send-tasks/:id/mark-sent-current-window` 已禁用。它们不会再把任务直接改成 `sent`；所有发送完成状态都必须来自发送适配器结果，真实桥接发送还必须等待 `bridge-ack`。

预留适配器：`windows_bridge`

- 可以通过 `WECHAT_SEND_ADAPTER=windows_bridge` 启用。
- 未接入真实 Windows 桥接程序前，它只会在 `.runtime/wechat-outbox` 生成待桥接 JSON 文件。
- 生成 outbox 后，发送任务会保持 `sending`，发送尝试会保持 `started`，表示“等待桥接回执”。
- 外部桥接程序完成后，调用 `POST /api/wechat/send-tasks/:id/bridge-ack` 回传 `sent` 或 `failed`。
- 回执成功后，任务更新为 `sent`；如果关联的是报价任务，报价也会回写为 `sent`。
- 回执失败后，任务更新为 `failed`；如果关联的是报价任务，报价会回到 `manual_review`，提醒人工处理。
- 工作台不提供手动“桥接成功回执”按钮，避免人工误把未真实发送的任务标记为 `sent`；人工处理异常时应取消任务、重新排队或扫描 worker/inbox 回执。

桥接程序也可以走文件回执：

```http
GET /api/wechat/bridge/outbox
GET /api/wechat/bridge/status
POST /api/wechat/bridge/inbox/scan
```

- `bridge/outbox` 只返回仍处于 `sending`，且最新 `windows_bridge` attempt 仍是 `started` 的任务。
- `bridge/status` 汇总 worker 最近一次运行状态、outbox 待处理数、inbox 待扫描回执数和账号锁状态。
- 已完成、失败、取消或旧 outbox 文件会进入 `ignored`，不会被桥接程序误处理。
- outbox 列表会返回 `preview` 摘要，展示目标微信账号、客户会话、动作数量、文字长度、图片数量、窗口快照和 outbox 文件名；列表不直接暴露完整原始 payload。
- 新版 outbox 文件包含 `version: wechat_bridge_outbox_v1`、一次性 `ackToken`、`target` 和 `sendPlan`。真实桥接器应按 `sendPlan.actions` 顺序执行文字/图片动作，并继续校验 `target` 里的账号、会话和客户身份。
- 桥接 worker 和后端 `bridge-ack` 在写入成功回执前都会校验 API pending 摘要和本地 outbox JSON 本体，确认文件路径、协议版本、账号、会话、动作数量、安全约束和发送守卫都匹配，不能只凭 `taskId` 或文件名伪造成功。
- 桥接程序可以把回执 JSON 放到 `.runtime/wechat-inbox`，字段包括 `version: wechat_bridge_ack_v1`、`ackToken`、`taskId`、`attemptId`、`wechatAccountId`、`conversationId`、`outboxFileName`、`status`、`errorMessage`、`metadata`、`sentAt`。
- `bridge/inbox/scan` 会读取回执，复用同一套 `bridge-ack` 逻辑，并把回执文件归档到 `processed` 或 `failed` 子目录。
- `bridge/status` 和 `bridge/inbox/scan` 只返回回执摘要，例如任务、attempt、状态、文件名和 `hasAckToken`，不会返回 `ackToken` 本体或原始回执 `data`。
- 详细协议见 `docs/WECHAT_BRIDGE_PROTOCOL.md`。真实桥接程序必须按该协议接入，不能只凭 `taskId` 生成成功回执。

## Windows 桥接 worker 骨架

新增脚本：

```bash
npm run wechat:bridge:once
npm run wechat:bridge:watch
```

默认模式是 `BRIDGE_MODE=noop`：

- 只读取 `GET /api/wechat/bridge/outbox`。
- 按微信账号创建本地锁，避免一个账号同时处理多个发送任务。
- 不写回执，不会把任务标记成已发送。

开发演示模式：

```bash
BRIDGE_MODE=simulate_sent npm run wechat:bridge:once
BRIDGE_MODE=simulate_failed npm run wechat:bridge:once
```

Windows PowerShell 可用：

```powershell
$env:BRIDGE_MODE='simulate_sent'; npm.cmd run wechat:bridge:once
```

配置项：

- `BRIDGE_API_BASE`：默认 `http://127.0.0.1:3200/api`
- `BRIDGE_MODE`：`noop`、`simulate_sent`、`simulate_failed`
- `BRIDGE_ACK_TRANSPORT`：`file_scan`、`file`、`api`
- `WECHAT_BRIDGE_INBOX_DIR`：默认 `.runtime/wechat-inbox`
- `WECHAT_BRIDGE_LOCK_DIR`：默认 `.runtime/wechat-bridge-locks`
- `WECHAT_BRIDGE_WORKER_STATUS_FILE`：默认 `.runtime/wechat-bridge-worker-status.json`
- `BRIDGE_LIMIT`：单轮最多处理几个 outbox 任务

这只是桥接骨架。真实微信 PC 发送器只能接在 worker 内部的发送动作位置，并且必须继续使用当前任务里的账号、会话、窗口校验结果和单账号锁；不能绕过平台登录、不能跨客户会话发送、不能把未真实发送的任务伪造成成功。

后续如果要接真实微信 PC 操作，应该只替换或扩展适配器层，继续复用现有的账号校验、聊天对象校验、最近消息校验、单账号串行锁和发送审计。

## 发送失败后的人工处理

发送队列支持人工重新排队和取消：

- `POST /api/wechat/send-tasks/:id/requeue`
- `POST /api/wechat/send-tasks/:id/cancel`

适用场景：

1. 当前微信窗口错聊导致任务被 `blocked`，客服切回正确窗口后可以重新排队。
2. Windows 桥接发送失败后，客服可以修复问题再重新排队。
3. 客户临时撤回需求或客服确认不再发送时，可以取消任务。
4. 如果取消的是等待桥接回执的任务，系统会把那条 `started` 的桥接发送尝试关闭为 `failed`，避免审计记录一直挂起。
5. 如果任务关联报价，取消或桥接失败会把报价转回 `manual_review`；重新排队会把报价恢复到 `send_queued`。

## 发送异常扫描

发送队列支持人工扫描异常：

- 前端按钮：发送安全队列里的 `扫描异常`
- API：`POST /api/wechat/send-tasks/scan-ops`

扫描会处理这些情况：

1. `windows_bridge` 任务长时间停在 `sending`，并且发送尝试仍是 `started`，说明桥接程序没有回执。系统会把任务转为 `failed`，关闭 pending attempt，并生成提醒。
2. 任务长时间停在 `queued`，系统会生成“排队过久”提醒。
3. 任务处于 `blocked` 或 `failed`，系统会生成一次站内提醒，避免客服漏看。
4. 同一个状态不会重复刷提醒；任务重新排队或状态变化后，下一次异常会重新提醒。

默认配置：

- `SEND_BRIDGE_ACK_TIMEOUT_MINUTES=5`
- `SEND_QUEUE_STALE_MINUTES=10`

## 真实商品库

现在商品库不再只保存演示字段，导入时可以带上真实销售和交付要素：

```text
SKU编号、商品名称、商品类型、分类、成本价、售价、库存、场景标签、主图、多角度图、尺寸、重量g、材质、供应商、交期天数、替代SKU、搭配规则
```

用途说明：

1. `主图` 和 `多角度图`：后续提交设计任务时，用真实 SKU 图片约束出图，避免设计平台凭空换包装。
2. `成本价`、`售价`、`库存`：用于预算搭配、利润计算和库存不足替代。
3. `场景标签`：让系统知道商品适合员工福利、客户拜访、节日礼赠、活动伴手礼等场景。
4. `尺寸`、`重量g`、`材质`、`交期天数`：给客服判断包装、物流、交付风险用。
5. `供应商`：后续做采购、补货、售后追责时可以追到来源。
6. `替代SKU`：库存不足时优先推荐可替代商品。
7. `搭配规则`：可以先写文字备注，也可以写 JSON，后续扩展禁配、必配、优先搭配规则。

商品库新增体检接口：

```http
GET /api/catalog/skus/audit
```

体检会提示这些问题：

1. 严重问题：缺 SKU 编号、缺商品名称、售价无效。
2. 警告问题：成本高于售价、缺主图、缺供应商、缺场景标签、库存为 0。
3. 提醒问题：低库存、缺尺寸、缺重量。

工作台里 `商品资料问题` 指标和商品库体检面板会直接显示这些问题。真实上线前，至少要让常卖 SKU 有主图、售价、成本、库存、供应商、场景标签、尺寸和重量。

商品库运营接口：

```http
GET /api/catalog/skus?includeInactive=true
POST /api/catalog/skus
POST /api/catalog/skus/import-preview
POST /api/catalog/skus/import-file-preview
POST /api/catalog/skus/import-file
GET /api/catalog/skus/import-fields
GET /api/catalog/skus/import-template
POST /api/catalog/skus/bulk
POST /api/catalog/skus/batch-update
POST /api/catalog/skus/:skuCode/deactivate
POST /api/catalog/skus/:skuCode/restore
GET /api/catalog/skus/change-logs
POST /api/assets/upload
```

用途说明：

1. `includeInactive=true`：查看已经下架的商品，方便恢复或复盘历史搭配。
2. `POST /api/catalog/skus`：手动新增或编辑单个 SKU。
3. `import-preview`：只解析粘贴的 Excel/CSV 文本，不保存，返回可导入商品、错误行和商品资料体检结果。
4. `import-file-preview`：解析上传文件的 base64 内容，支持 `.xlsx`、`.csv`、`.tsv`、`.txt`，不保存。
5. `import-file`：直接解析文件并入库，工作台默认仍建议先预览再确认。
6. `import-fields`：返回标准导入字段、必填项、示例值和可识别别名，前端用它展示字段说明。
7. `import-template`：返回标准 `.xlsx` 模板的 base64 内容，默认格式为 `xlsx`，也支持 `?format=csv`。
8. `bulk`：确认预览无误后批量入库。
9. `batch-update`：批量调整库存、售价、供应商、交期、场景标签或上下架状态。
10. `deactivate`：下架商品。下架后不会参与自动搭配，但保留历史数据。
11. `restore`：恢复商品。适合补货后重新进入自动推荐。
12. `change-logs`：查看 SKU 最近变更，支持按 `skuCode` 过滤，方便追踪谁改了库存、售价、主图、上下架状态等关键字段。
13. `assets/upload`：复用素材上传能力保存 SKU 主图或多角度图，`ownerType` 使用 `sku`，`ownerId` 使用 SKU 编号。

工作台商品库区域已经接入这些动作：可以勾选当前列表 SKU，批量修改库存/售价/供应商，也可以批量下架或恢复。单条 SKU 可以编辑、下架和恢复。商品表单支持上传本地图片，上传后会把本地素材路径写入主图或多角度图字段；导入区支持下载标准 Excel 模板、粘贴表格或选择文件先预览、看表头识别结果、错误行和缺图/低库存/利润异常，再确认入库。

商品体检现在会返回 `repairQueue` 补齐队列。队列会把缺 SKU 编号、缺名称、售价无效、成本高于售价、缺主图、主图文件失效、缺供应商、缺场景、库存为 0、缺尺寸、缺重量等问题按优先级排序。工作台里的“商品资料补齐向导”会显示最需要处理的 SKU，点 `补齐` 会直接定位到该商品编辑表单，避免运营在大表里手动找问题。

商品图片检查会覆盖主图和已填写的多角度图：本地图片路径不存在会提示重新上传，`.pdf`、`.txt` 等明显不是图片的路径会提示格式异常。本地图片还会读取文件头，避免把改名成 `.jpg` 的文本文件当成真实商品图。远程 `http/https` 图片不会做在线探测，避免因为网络波动误判；如果远程链接明确带有非图片后缀，也会在体检中提示。

图片体检还会返回 `imageProblems` 清单，里面会标明 SKU、商品名、问题类型、图片字段、主图/多角度图、第几张多角度图和原始路径。工作台里的“图片问题清单”会直接展示这些信息，点击 `编辑图片` 会定位到对应商品表单；点击 `移除路径` 只会先修改当前表单，不会直接保存，运营确认后再点 `保存商品` 生效。图片问题支持导出 CSV，表里包含严重程度、处理建议和原始路径，适合交给运营按清单补拍、重传或确认删除失效路径。

SKU 新增、手动保存、批量导入、批量修改、下架和恢复都会写入变更日志。日志会记录变更来源、操作人、变更字段、变更前和变更后的值；本地 JSON 模式默认保留最近 1000 条。工作台商品库区域会显示最近变更，便于客服或运营发现“库存为什么变了”“价格是谁改的”“主图是否刚被替换”。

## 设计平台稳定性演示

工作台顶部和设计任务操作区已经接入设计平台稳定性能力：

1. 点击 `检测设计平台`，客服平台会请求 `GET /api/integrations/design-platform/health`，确认设计平台是否在线，并显示延迟。
2. 点击 `失败演示`，系统会创建一条失败设计任务。
3. 选中失败任务后点击 `重试`，系统会重新提交到设计平台，生成新的 `externalJobId`。
4. 点击 `轮询结果`，系统会主动请求设计平台结果；如果已完成，会把 4-6 张候选图绑定回正确客户、会话和任务。
5. 点击 `取消`，如果任务已经提交到设计平台，会同步调用设计平台取消接口，并把本地任务标记为 `cancelled`。
6. 如果设计平台回调 `failed`，系统会自动重试 1 次；第二次仍失败时保留失败状态并提醒客服人工处理。

这一块对应正式商业落地里的“设计平台不稳定也不能让客服流程断掉”：健康检查、失败重试、人工轮询、取消、提醒都已经有基础闭环。

## 客户改图与修改次数演示

聊天侧栏里有 `客户改图`：

1. 客户选图后，把客户的修改要求粘贴到输入框，例如“Logo 放大一点，背景换成浅色，商品摆放更商务”。
2. 点击 `提交改图`，系统会先记录一条修改请求，包含第几次修改、客户原话、选中的候选图和处理策略。
3. 低预算客户默认前 2 次免费改图会自动重新提交到现有设计平台 `/v1/design-jobs`，不新增外部接口。

这块对应商业落地里的“首轮 + 2 轮修改、超出收费或人工处理”。后续可以把免费次数、收费规则和高价值阈值做成后台配置。

## 客户选图识别演示

聊天侧栏候选图区域支持三种选图方式：

1. 文字选择：客户说“第 2 张”“我选 3 号”“pick 2”，系统会按候选图顺序匹配。
2. 引用图片：客户在微信里引用某张候选图回复时，系统可用候选图 ID 直接匹配，不依赖文字。
3. 截图匹配：候选图保存时会生成 `fingerprint`，客户发回截图时可用截图指纹匹配候选图。
4. 如果截图相似度不足，系统不会硬猜，会创建提醒并进入人工确认。

当前本地演示使用稳定哈希作为图片指纹；正式商业部署可以换成真实图片感知哈希或设计平台返回的图片 ID/文件哈希。

## 人工审核中心演示

页面中部有 `人工审核中心`，用于处理高价值客户和异常任务：

1. 高价值设计任务、失败任务、超时任务会进入 `待审设计`。
2. 人工可以选择 `通过`、`批准发送`、`要求改图`、`驳回`。
3. `通过` 会把设计任务从 `manual_review` 改为 `quick_confirm`，等待客服快速确认。
4. `批准发送` 会进入发送安全队列，仍然要经过微信账号、聊天对象、最近消息/客户 ID 三重校验。
5. 高价值报价会进入 `待审报价`，支持 `通过报价`、`继续跟进`、`驳回报价`。
6. 每次审核都会生成 ReviewLog，记录审核人、决策、前后状态和备注，方便后续复盘。

## 微信窗口快照演示

`多账号控制` 里每个账号可以模拟三种窗口状态：

1. `正确窗口`：当前微信账号、聊天对象、最近客户消息都和发送任务匹配。
2. `错聊窗口`：当前窗口停留在另一个客户，系统会拦截发送。
3. `离线窗口`：当前账号窗口不可用，系统会要求人工确认。

`发送安全队列` 支持 `当前快照校验` 和 `执行当前适配器`。快照只负责证明“现在是不是正确微信账号、正确聊天对象、正确客户上下文”，不能直接代表消息已经发送。正式接入微信 PC 时，只需要把真实窗口识别结果写入同一个快照接口，后面的安全闸门可以复用。

默认只有 30 秒内采集的窗口快照可以用于真实发送校验，避免用旧窗口状态误把 A 客户的内容发给 B 客户。需要调整时可设置 `WECHAT_WINDOW_SNAPSHOT_MAX_AGE_SECONDS=30`。

外部窗口识别器也可以走文件接入箱，不需要直接调用数据库：

```http
POST /api/wechat/window-snapshots/inbox/scan
```

默认扫描目录：

```text
.runtime/wechat-window-snapshots
```

识别器每次写一个 `.json` 文件即可，扫描成功后文件会被移到 `processed`，失败会移到 `failed`。示例：

```json
{
  "source": "windows_observer",
  "isOnline": true,
  "wechatAccountId": "wechat_demo_1",
  "accountDisplayName": "客服微信 1",
  "activeChatTitle": "王总 端午礼盒",
  "chatTitle": "王总 端午礼盒",
  "recentCustomerId": "customer_demo_1",
  "recentMessageText": "客户最近一句消息",
  "confidence": 0.96
}
```

也支持批量文件：

```json
{
  "snapshots": [
    {
      "source": "windows_observer",
      "isOnline": true,
      "wechatAccountId": "wechat_demo_1",
      "chatTitle": "王总 端午礼盒",
      "recentCustomerId": "customer_demo_1",
      "confidence": 0.96
    }
  ]
}
```

这层只接收窗口状态和最近消息证据，不做微信登录、多开或绕过平台限制的能力。

### Windows 前台窗口观察器

新增本地观察器脚本：

```bash
npm.cmd run wechat:observe-window:once
npm.cmd run wechat:observe-window:watch
```

默认行为：

- 只读取当前前台窗口标题、进程名、进程 ID。
- 如果识别为微信窗口，就写入 `.runtime/wechat-window-snapshots/*.json`。
- 每次运行都会写状态文件 `.runtime/wechat-window-observer-status.json`。
- 不点击、不输入、不发送、不登录、不多开。
- 默认不自动扫描入库；如果要写完立刻让后端扫描，直接运行：

```bash
node tools/wechat-window-observer.js --once --scan
```

建议先创建配置文件：

```text
.runtime/wechat-window-observer-config.json
```

示例：

```json
{
  "accounts": [
    {
      "name": "客服微信1",
      "wechatAccountId": "wechat_demo_1",
      "accountDisplayName": "客服微信 1",
      "processNames": ["WeChat"],
      "titleIncludes": ["王总"]
    }
  ],
  "conversations": [
    {
      "name": "王总礼盒会话",
      "wechatAccountId": "wechat_demo_1",
      "conversationId": "conversation_demo_1",
      "customerId": "customer_demo_1",
      "titleIncludes": ["王总"],
      "chatTitle": "王总 端午礼盒"
    }
  ]
}
```

匹配逻辑很简单：

- `processNames`：窗口进程名要匹配，比如 `WeChat`。
- `titleIncludes`：窗口标题必须包含这些词。
- `titleAnyIncludes`：窗口标题包含其中任意一个词即可。
- `titleEquals`：窗口标题必须完全相等。

如果没有匹配到账号或客户，会照样写快照，但后端诊断会把它判为不安全，发送队列不会通过。

工作台 `多账号控制` 面板会显示观察器状态：

- 最后一次观察是多久前。
- 当前前台进程名。
- 是否识别到微信窗口。
- 是否匹配到微信账号。
- 置信度、是否 dry-run、是否自动扫描。
- `采集当前窗口` 按钮会让后端运行一次观察器，写入快照文件后立即扫描入库。

也可以直接调用接口查看：

```http
GET /api/wechat/window-observer/status
```

也可以直接触发一次采集：

```http
POST /api/wechat/window-observer/capture-once
```

## 发送执行审计演示

`发送安全队列` 里新增 `执行干跑发送`：

1. 系统先用最近一次微信窗口快照做发送前校验，并要求快照足够新。
2. 校验失败时，任务会被拦截，并生成一条失败尝试记录。
3. 校验通过时，系统会创建一条 `dry_run` 发送尝试记录，并把任务状态改成 `dry_run`。
4. 这里不会真实发送微信消息，只记录 payload 摘要、文本长度、图片数量、窗口快照 ID、校验状态和错误原因；商业真实发送要重新排队后使用 `windows_bridge` 并等待回执。
5. 后续接真实微信发送器时，可以复用同一套发送尝试审计表，便于排查“为什么没发、发给谁、哪个窗口状态下发的”。

如果使用 `windows_bridge` 适配器，任务不会直接变成已发送：

1. 系统写入 outbox 文件，并把任务状态改为 `sending`。
2. 桥接 worker 必须回传 `attemptId`、微信账号、会话 ID 和 `outboxFileName`。
3. 后端会校验回执和 pending attempt、outbox 文件是否匹配。
4. 成功回执匹配后才标记 `sent`；失败回执会标记 `failed` 并转人工。
5. 处理完回执后，outbox 文件会归档到 `processed`、`failed` 或 `cancelled`，避免重复处理旧发送任务。
6. `sending` 状态不能直接重新排队，必须等待回执、扫描超时或取消任务。

## 安全处理发送队列

新增接口：

```http
POST /api/wechat/send-tasks/process-safe-queue
```

用途：批量处理待发送任务，但不是直接群发。系统会按微信账号分组，每个账号一次只处理队首任务。

处理顺序：

1. 只扫描 `queued` 状态任务。
2. 每个微信账号只取最前面的一个任务。
3. 读取该账号最新微信窗口快照。
4. 校验微信账号、当前聊天对象、最近消息/客户 ID、单账号队列顺序、窗口快照新鲜度。
5. 校验通过后，交给当前发送适配器。
6. 校验失败时，任务变成 `blocked`，并记录失败原因和发送尝试。

工作台 `发送安全队列` 区域新增 `安全处理队列` 按钮。默认 `dry_run` 适配器只做校验和审计，不会真实操作微信；后续切换 `windows_bridge` 后，也仍然要等待桥接程序回执，不会假装发送成功。

## 已完成

- 设计平台 API 契约和回调入口。
- 礼盒效果图任务状态流。
- 预算解析：每份预算、总预算+数量。
- 高价值规则：订单总额或单份金额达到 10000 元转人工。
- SKU 搭配：按预算、场景、库存、替代品生成组合。
- 客户选图识别：支持“第3张/第二个/4号”等文字形式。
- 聊天记录导入：从 `客户：/客服：` 文本中提取训练样本。
- Agent/Skill 中心：按售前、礼盒设计、物流、售后、尺码等场景拆分能力。
- Skill 生成器：从高分聊天样本提炼预算澄清、设计需求确认、物流安抚、高情商话术等可复用技能，并对同名 Skill 做版本更新。
- Skill 增强回复：路由评估会读取 Agent Skill 和历史知识样本，生成更像人工客服的回复草稿，并展示命中的 Skill、参考样本和安全检查。
- 入站消息自动处理：客户消息进入后会落库、路由、生成回复草稿、必要时创建设计草稿或转人工，发送仍走安全队列。
- 设计草稿自动提交：低风险且资料齐全的设计草稿可以批量提交到设计平台，缺资料或高价值任务自动跳过。
- 高价值转人工：入站消息、直接建设计任务、客户选图、手动转人工和历史任务扫描都会把高价值任务保留给人工处理，并锁定会话、记录 ReviewLog。
- 低价值自动处理：低价值设计草稿自动提交，已出图的低价值任务自动进入微信安全发送队列。
- 知识沉淀模型：本地 JSON 和 Prisma schema 都预留训练样本、知识条目和导入批次。
- 多微信账号演示数据：微信客服1号和微信客服2号各自绑定独立客户会话。
- 发送队列安全闸门：账号、聊天对象、最近消息/客户 ID、单账号串行队列校验。
- 发送任务入队绑定校验：任务创建时校验微信账号、会话、客户、设计任务和报价草稿，错绑任务不能进入队列。
- 安全处理发送队列：按微信账号只处理队首任务，校验通过才执行适配器，失败会拦截并留审计记录。
- Windows 桥接 outbox/inbox：可列出待桥接任务，接收文件回执并归档，完成发送状态闭环。
- 微信窗口快照适配层：支持记录账号窗口、当前聊天对象、最近客户标识、离线状态，支持文件接入箱扫描真实识别器快照，并提供 Windows 前台窗口观察器，用最近快照执行发送前校验，且默认拦截超过 30 秒的旧快照。
- 发送执行审计：支持 dry-run 发送尝试记录、拦截记录、payload 摘要、窗口快照绑定、bridge attempt/outbox 回执绑定和最近尝试展示。
- 智能体路由决策：根据客户消息识别场景、预算、价值等级、缺失信息和人工介入条件。
- 报价/订单草稿：客户选图后生成报价，展示总价、成本、利润、付款状态和跟进人。
- 客户选图自动报价：低价值任务选图明确后自动生成报价草稿，高价值任务继续转人工审核。
- 商品导入与搭配：支持 Excel 复制表格/CSV 文本导入 SKU，并按预算推荐礼盒组合。
- 真实商品库字段与体检：支持主图、多角度图、尺寸、重量、材质、供应商、交期、替代 SKU 和搭配规则，并通过 `/api/catalog/skus/audit` 检查缺图、低库存、缺供应商、成本倒挂等问题。
