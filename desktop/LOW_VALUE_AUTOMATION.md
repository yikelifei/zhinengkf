# 低价值客户自动处理说明

这个文件说明“低价值自动处理”现在是怎么跑的，方便后续接真实微信发送器时继续扩展。

## 它负责什么

后台自动化会每隔一段时间跑一轮，默认 15 秒一次：

1. 轮询还在出图中的设计任务。
2. 找到信息完整、低价值、未提交的设计草稿，自动提交到设计平台。
3. 找到已经出图完成、低价值、可快速确认的任务，把候选图放进微信安全发送队列。
4. 客户选图后，如果报价草稿属于低价值且信息完整，自动放进微信安全发送队列。
5. 报价已发送、客户已接受或已标记定金/付款后，低价值报价自动生成订单草稿。
6. 客户明确接受报价、确认下单、付定金或付款后，低价值订单确认回复会进入微信安全发送队列。
7. 客户在微信里确认报价或付款时，自动生成订单草稿，并把一条自然确认回复放入微信安全发送队列。
8. 扫描出图超时、发送队列超时，生成提醒。
9. 只处理带有低价值自动化标记的发送任务。

高价值客户、需要人工审核、缺少素材、缺少微信会话绑定的任务不会自动发送。

## 安全边界

低价值自动发送仍然遵守微信账号队列顺序：

- 同一个微信账号一次只处理一个发送任务。
- 如果账号前面还有更早的待发任务，低价值图片不会插队。
- 发送前仍然要经过账号、当前聊天对象、最近消息或客户 ID 校验。
- 高价值任务即使误进队列，也会被低价值自动化过滤掉。
- 低价值报价任务也必须带有自动化标记，后台才会尝试安全发送。

这样做的目的不是为了快，而是避免把 A 客户的图发给 B 客户。

## 自动发送暂缓怎么处理

低价值任务进入发送队列后，如果同一个微信账号前面还有更早的待发任务，系统不会让它插队。后台会做两件事：

- 在自动化结果里返回 `advice`，说明为什么跳过、卡在哪个前序任务、建议怎么处理。
- 只提醒一次客服“低价值自动发送被前序任务卡住”，避免每 15 秒重复刷提醒。

客服看到这类提醒时，处理顺序是：

1. 先看前序发送任务是不是还应该发。
2. 如果应该发，先处理前序任务。
3. 如果不应该发，取消前序任务。
4. 再点“后台跑一轮”或等待下一轮自动化，低价值任务会继续安全发送。

## 常用接口

查看自动化状态：

```http
GET http://127.0.0.1:3200/api/automation/status
```

手动跑一轮：

```http
POST http://127.0.0.1:3200/api/automation/run-once
```

前端工作台：

```text
http://127.0.0.1:3100/
```

## 环境变量

这些配置都可以后续放进启动脚本或 `.env`：

```env
LOW_VALUE_AUTOMATION_ENABLED=true
LOW_VALUE_AUTOMATION_RUN_ON_START=true
LOW_VALUE_AUTOMATION_INTERVAL_MS=15000
LOW_VALUE_AUTOMATION_POLL_LIMIT=50
LOW_VALUE_AUTOMATION_PROCESS_SEND_QUEUE=true
LOW_VALUE_AUTOMATION_SEND_QUEUE_LIMIT=10
```

字段意思：

- `LOW_VALUE_AUTOMATION_ENABLED`：是否开启后台自动化。
- `LOW_VALUE_AUTOMATION_RUN_ON_START`：API 启动后是否先自动跑一轮。
- `LOW_VALUE_AUTOMATION_INTERVAL_MS`：每隔多少毫秒跑一轮，默认 15000。
- `LOW_VALUE_AUTOMATION_POLL_LIMIT`：每轮最多轮询多少个设计任务。
- `LOW_VALUE_AUTOMATION_PROCESS_SEND_QUEUE`：是否让后台自动处理低价值安全发送队列。
- `LOW_VALUE_AUTOMATION_SEND_QUEUE_LIMIT`：每轮最多检查多少个低价值发送任务。

## 客户选图后的自动报价

当客户在微信入站消息里明确说“选第 2 张”“就这个”“按这张报价”时，系统会优先查当前会话里最新的 `sent` 设计任务：

- 能高置信匹配到候选图：绑定选中图，低价值任务自动生成报价草稿。
- 报价草稿完整且未入队：后台低价值自动化把报价消息放入微信安全发送队列。
- 已经入队或已发送过报价：转人工，避免客户改选后重复报价。
- 高价值客户或总额达到高价值线：转人工报价。
- 没有可匹配任务、候选图缺失、截图/引用图不确定：转人工确认。

这一步不会绕过发送校验，报价发送仍然要经过微信账号、当前聊天对象、最近消息/客户 ID 的三重校验。

## 低价值订单草稿自动生成

低价值报价不会一生成就直接变订单，必须满足下面任一条件：

- 报价已经通过安全发送流程，状态为 `sent`。
- 客服或系统已经把报价标记为客户接受，状态为 `accepted`。
- 已经标记定金或已付款，付款状态为 `deposit_paid` 或 `paid`。

同时还必须满足：

- 已经有客户选中的效果图。
- 有客户、会话、微信账号和设计任务绑定。
- 总金额没有达到高价值线。
- 利润不是负数。
- 当前会话没有被人工接管。
- 这个报价还没有生成过订单草稿。

生成的订单草稿会保存报价、客户、会话、微信账号、礼盒组合、数量、价格、成本、利润和选中效果图快照，后续用于排产、收款和人工跟进。

报价只是已经发给客户时，系统只生成订单草稿，不会主动发“订单确认”给客户。只有客户明确接受报价、确认下单、付定金或付款后，系统才会生成一条不像模板的确认回复，并放入微信安全发送队列。例如：

- 未付款确认：先锁定方案，再逐项确认付款方式、交期和细节。
- 已付定金：记录定金，继续确认剩余款项和交期。
- 已付款：记录已付款，推进订单信息和排产。

这条回复会带上低价值自动化标记，后台自动队列才会处理。它不会绕过微信发送校验，仍然必须通过账号、当前聊天对象、最近消息/客户 ID 校验；如果已经有确认发送任务，系统会跳过，避免重复发给客户。

## 当前限制

现在仍然是本地演示模式：

- 微信发送器还没有真正接入微信 PC 操作。
- 发送队列会进行安全校验和状态流转，但不会真正把消息发到微信。
- 真实商业落地时，需要接入微信窗口识别、焦点锁、当前聊天对象校验和发送结果回执。
