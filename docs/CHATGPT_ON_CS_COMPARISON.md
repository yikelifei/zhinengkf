# ChatGPT-On-CS-main 对比与可借鉴方案

参考项目：`ChatGPT-On-CS-main` 本地对比副本。

## 总结结论

当前项目更适合作为你的主项目继续做。

`ChatGPT-On-CS-main` 更像一个通用电商客服桌面框架，优势在多平台接入、插件、关键词 Excel 管理、随机话术、模型供应商兼容。你的 `zhinengkefu` 更像一个面向礼盒定制业务的成交型客服系统，已经有客户资料、报价准备度、高价值客户筛选、人工交接、跟进报表、知识库和客服风格控制。

所以不建议整体替换。更合适的做法是：保留当前项目的业务内核，把对方项目的成熟产品能力分批吸收。

| 维度 | 当前项目 zhinengkefu | ChatGPT-On-CS-main |

| 技术栈 | Python、SQLite、YAML、Web 控制台、微信 PC 自动化 | Electron、React、TypeScript、Express、Socket.IO、Sequelize |
| 定位 | 礼盒定制成交客服、线索筛选、报价和交付闭环 | 通用电商客服桌面工具 |
| 平台能力 | 当前主要围绕微信 PC | 抽象了微信、千牛、抖音、拼多多、小红书、京东等平台 |
| 知识库 | YAML 知识库、规则、AI 兜底、业务守卫 | README 提到知识库和聊天记录学习，代码核心更多是关键词、GPT、插件 |
| 客户筛选 | 已有线索管道、高价值客户评分、人工交接 | 更偏消息和平台会话管理，业务评分不明显 |

| 管理体验 | Web 控制台正在完善 | 桌面 UI 更完整，Excel 导入导出更成熟 |

## 对方值得借鉴的能力

1. 多话术随机表达

适合直接吸收。客户不能每次收到一模一样的模板回复，否则很像机器人。当前已加入：

- `[or]`：同一条规则可以写多个回复版本，系统随机选一个。
- `[~]`：在模板作者指定位置加入轻语气词，让短句更自然。

示例：

```text
可以的[~]，您这边大概需要多少份？[or]没问题[~]，我先帮您看下数量和预算。
```

2. 平台适配层

值得吸收，但不要立刻大改。当前微信自动化逻辑应逐步封装成 `ChannelAdapter`：

- `receive_messages()`
- `send_text()`
- `send_image()`
- `send_file()`
- `get_contact_profile()`

后续接企业微信、网页客服、抖音、其他软件 API 时，只新增适配器，不改客服大脑。

3. 插件机制

可以借鉴思路，但不要照搬 JS 任意插件。对方插件能访问 `fs`、`axios` 等能力，扩展强但风险也高。

当前项目更适合做白名单工具插件：

- 查询订单
- 查询报价
- 创建出图任务
- 生成提示词
- 标记高价值客户
- 推送人工交接

所有插件必须有输入输出 schema、日志、超时和权限控制。

4. Excel 导入导出

值得做。客服人员更容易维护 Excel，而不是直接改 YAML。

优先支持：

- 知识库导入导出
- 常见问题导入导出
- 人工交接关键词导入导出
- 禁用词和敏感词导入导出
- 高价值客户报表导出

5. 上下文字段扩展

对方项目对电商字段提取得比较细，例如商品、会员标签、新客、订单金额、物流状态。

你的业务应该扩展为：

- 客户类型：企业采购、渠道、个人、老客
- 需求品类：礼盒、粽子、月饼、年货、伴手礼
- 数量区间
- 预算区间
- 交付日期
- 是否需要设计
- 是否需要开票
- 是否需要寄样
- 出图任务状态
- 报价状态

6. 模型供应商健康检查

可借鉴，但优先级低于业务闭环。当前应先保证一个主模型可稳定回复，再做多模型自动切换。

## 不建议直接照搬的部分

1. 不建议直接复制代码

该项目使用 AGPL-3.0 许可证。如果你的软件以后要商业化，直接复制核心代码可能带来开源披露风险。可以学习架构和产品思路，然后在当前项目里重新实现。

2. 不建议先切 Electron

Electron UI 更像完整桌面软件，但切栈成本高。你当前最重要的是客服效果、线索筛选、知识库、交付流程，不是先换壳。

3. 不建议放开任意插件权限

客服系统会接触客户资料、聊天记录、订单和报价。插件不能随便读写本机文件或访问网络，必须做权限和审计。

## 建议实施顺序

第一阶段：提升当前系统可用性

- 完善今日工作台
- 完善高价值客户筛选
- 完善话术随机表达
- 报价准备度和人工交接稳定运行
- 导出日报、线索报表、高价值客户报表

第二阶段：知识库和聊天记录学习

- 微信聊天记录导入
- 自动提取常见问答
- 自动提取客户异议
- 自动提取高转化真人话术
- 人工审核后进入知识库

第三阶段：出图交付闭环

- 从客户需求中提取出图提示词
- 创建出图任务
- 记录生成结果
- 客户要求修改时进入人工队列
- 客户确认后归档案例

第四阶段：开放 API

- 提供 `/api/chat/reply`
- 提供 `/api/leads/high-value`
- 提供 `/api/image-jobs`
- 提供 `/api/knowledge/search`
- 给其他软件调用，但必须带鉴权、限流和日志

第五阶段：多平台适配

- 把微信 PC 自动化改造成一个适配器
- 再扩展企业微信、网页客服或其他平台
- 保持客服大脑不变，只替换消息入口和发送出口

## 当前已吸收的一项能力

已把对方项目的随机话术思路实现到当前模板渲染器：

- 文件：`core/template_renderer.py`
- 测试：`tests/test_core.py`

这样规则回复可以从“固定模板”变成“多版本自然表达”，更接近真人客服，同时不破坏现有规则引擎。
