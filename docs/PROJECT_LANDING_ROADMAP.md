# 智能客服项目落地总规划

本文档用于后续开发、复盘和让 AI 助手快速理解项目方向。它是当前阶段的主规划文档。

更新时间：2026-06-24

## 1. 项目定位

本项目不是通用聊天机器人，也不是单纯的模板客服。

目标是做一个面向“礼盒定制、企业采购、设计出图交付”的成交型智能客服系统：

- 抖音、小红书等平台负责获客。
- 微信负责主要沟通、转化、报价和交付。
- 智能客服负责快速响应、收集需求、筛选高价值客户、生成出图提示词、辅助人工成交。
- 人工负责最终报价、异常处理、客户情绪处理、改图确认和订单交付。

核心原则：

- 不臆造平台接口。
- 不走高风险私信抓包或逆向接口。
- 不让 AI 自由承诺价格、交期和售后。
- 先做好自己的业务闭环，再考虑做成通用 SaaS。

## 2. 业务主链路

完整链路如下：

```text
抖音/小红书内容获客
        ↓
客户评论、私信、咨询
        ↓
平台内轻回复/人工筛选
        ↓
引导客户主动加微信或留下联系方式
        ↓
微信智能客服承接
        ↓
识别客户来源和需求
        ↓
收集报价关键字段
        ↓
高价值客户评分
        ↓
知识库/RAG/业务规则生成回复
        ↓
需要设计时提取出图提示词
        ↓
进入出图软件生成图片
        ↓
人工处理修改和确认
        ↓
报价、打样、生产、发货
        ↓
复盘聊天记录，沉淀知识库和高转化话术
```

## 3. 当前产品边界

### 必须做

- 微信自动回复和人工接管。
- 知识库回复，不能只套模板。
- 真人化表达，不机械、不暴露 AI。
- 高价值客户筛选。
- 报价信息补齐。
- 出图需求提取为提示词。
- 平台来源记录：抖音、小红书、微信、其他。
- 微信聊天记录导入后的知识库学习。
- 本地 Web 控制台统一管理。
- API 接口，供后续出图软件或其他软件调用。

### 暂不做

- 不做抖音、小红书私信全自动抓包。
- 不承诺普通账号能拿到私信 API。
- 不直接复制开源项目代码。
- 不先做多租户 SaaS。
- 不让 AI 自动成交大单或自动承诺最终报价。

## 4. 系统总架构

```text
获客平台层
  ├─ 抖音
  ├─ 小红书
  ├─ 其他内容平台
  └─ 手动录入/表格导入

渠道层
  ├─ 微信 PC Adapter
  ├─ 后续官方 API Adapter
  └─ ChannelHub 统一消息格式

客服大脑层
  ├─ 意图识别
  ├─ 业务规则
  ├─ 知识库/RAG
  ├─ 真人风格润色
  ├─ 安全护栏
  ├─ 高价值客户评分
  └─ 人工接管判断

业务工作流层
  ├─ 线索池
  ├─ 报价准备度
  ├─ 出图任务队列
  ├─ 跟进提醒
  ├─ 订单交接
  └─ 运营报表

接口层
  ├─ Web 控制台
  ├─ 本地 API
  ├─ 出图软件 API
  └─ 后续平台 API
```

## 5. 核心模块规划

### 5.1 渠道接入

当前状态：

- 微信已作为主渠道。
- 已新增 `ChannelHub` 多渠道基础结构。
- 抖音、小红书、拼多多、淘宝、快手目前是规划渠道，不能假装已接通。

下一步：

- 保持微信稳定。
- 增加“平台线索录入”，先记录抖音/小红书来源。
- 后续拿到官方接口或合规服务商接口后，再新增 Adapter。

### 5.2 平台获客与微信承接

目标：

让抖音、小红书客户最终沉淀到微信，在微信里完成深度沟通。

功能：

- 记录客户来源平台。
- 记录来源视频/笔记/关键词。
- 客户加微信后询问来源并绑定。
- 统计不同平台转化率。
- 维护不同平台的轻回复话术。

注意：

- 平台内不要生硬发送微信号。
- 优先用自然话术收集需求，再引导客户主动添加。
- 对明显高价值客户提醒人工介入。

### 5.3 知识库

知识库分三层：

1. 业务规则知识
   - 价格边界
   - 起订量
   - 交期
   - 发票
   - 打样
   - 物流
   - 禁止承诺

2. FAQ 问答知识
   - 客户常问问题
   - 标准答法
   - 多话术版本
   - 不同客户场景的变体

3. 案例知识
   - 成交案例
   - 礼盒方案
   - 设计风格
   - 节日场景
   - 客户异议和处理话术

后续微信聊天记录导入时，不直接全量塞进模型。流程应为：

```text
聊天记录导入
→ 清洗无效内容
→ 提取问答对
→ 提取高转化话术
→ 提取客户异议
→ 人工审核
→ 写入知识库
→ 回归测试
```

### 5.4 智能回复

回复策略：

```text
安全护栏
→ 意图识别
→ 明确业务问题：规则/知识库回答
→ 信息不完整：追问关键字段
→ 高价值客户：回复并提醒人工
→ 复杂报价/投诉/改图：人工接管
→ 低置信度：不乱答，转人工或追问
```

回复要求：

- 像真人客服。
- 简短、自然、有温度。
- 不一次性问太多。
- 不暴露“我是 AI”。
- 不编造价格、库存、交期。
- 不承诺“最低价”“当天发货”“一定能做”。
- 支持 `[or]` 多话术随机表达。

### 5.5 高价值客户筛选

评分维度：

- 数量
- 预算
- 交期
- 城市
- 企业采购
- 是否需要设计
- 是否需要开票
- 是否愿意加微信
- 是否反复追问
- 是否已有明确用途
- 来源平台

输出：

- 高价值客户列表
- 缺失字段
- 推荐下一步动作
- 是否需要人工接管
- 预计成交金额

### 5.6 出图任务

目标：

客户说出礼盒需求后，系统自动整理成出图软件可用的提示词。

流程：

```text
客户需求
→ 提取品类、风格、节日、颜色、文案、Logo、数量、用途
→ 生成提示词
→ 进入出图任务队列
→ 调用出图软件 API
→ 生成图片
→ 人工审核
→ 客户确认
→ 修改意见进入人工队列
```

AI 只负责提取和生成提示词。图片是否交付、是否修改，由人工最终确认。

### 5.7 人工接管

必须人工接管的场景：

- 投诉、退款、不满。
- 最终报价。
- 大额客户。
- 急单。
- 改图。
- 客户反复质疑。
- 需要明确承诺生产、物流或售后。
- 模型置信度低。

人工接管后：

- 自动回复暂停一段时间。
- 记录接管原因。
- 控制台显示待处理。
- 处理后可恢复自动回复。

### 5.8 API 接口

后续对外或给内部软件调用的接口：

- `/api/chat/reply`：根据消息生成建议回复。
- `/api/leads/import`：导入平台线索。
- `/api/leads/high-value`：获取高价值客户。
- `/api/knowledge/search`：搜索知识库。
- `/api/image-prompts/extract`：从客户需求提取出图提示词。
- `/api/image-jobs`：创建出图任务。
- `/api/channels`：查看渠道接入状态。

API 必须具备：

- 鉴权
- 日志
- 限流
- 错误返回
- 人工审核开关

## 6. 阶段性落地计划

### 第一阶段：微信客服闭环

目标：微信可稳定接待客户。

交付：

- 微信消息收发稳定。
- 知识库问答可用。
- 人工接管可用。
- 基础线索保存。
- Web 控制台可查看状态。
- 自动测试和 smoke 测试可运行。

验收：

- 常见问题能正确回复。
- 不乱报价。
- 投诉能转人工。
- 同一客户不会重复刷屏。

### 第二阶段：高价值客户和报价准备

目标：筛出真正值得跟进的客户。

交付：

- 高价值客户评分。
- 报价字段缺失提醒。
- 今日待办工作台。
- 跟进提醒。
- 高价值客户报表。

验收：

- 能识别数量、预算、交期、城市。
- 能区分普通咨询和高价值咨询。
- 能告诉人工下一步该问什么。

### 第三阶段：抖音/小红书获客承接

目标：让平台客户进入微信成交链路。

交付：

- 平台线索录入。
- 来源平台字段。
- 微信客户来源绑定。
- 平台引流话术库。
- 来源转化报表。

验收：

- 能记录客户来自哪个平台。
- 能统计平台带来的高价值客户。
- 微信对话能关联原始来源。

### 第四阶段：知识库学习

目标：让系统从真实客服聊天中变得更像真人。

交付：

- 微信聊天记录导入。
- 问答对提取。
- 客户异议提取。
- 高转化话术提取。
- 人工审核入库。

验收：

- 导入记录后不会污染知识库。
- 新知识必须审核。
- 回复质量有可复盘的样本。

### 第五阶段：出图交付

目标：把客户需求转成出图软件任务。

交付：

- 出图提示词提取器。
- 出图任务队列。
- 出图软件 API 对接。
- 修改意见记录。
- 案例沉淀。

验收：

- 客户需求能被结构化。
- 提示词能复用。
- 需要修改时进入人工队列。

### 第六阶段：稳定上线

目标：系统可长期使用。

交付：

- 启动脚本。
- 数据备份。
- 错误日志。
- 权限和 API Key 管理。
- 回归测试。
- 部署说明。

验收：

- 重启不丢数据。
- 接口异常不崩溃。
- 关键功能均有测试。
- 控制台能看到系统健康状态。

## 8. 风险和约束

### 平台风险

抖音、小红书不一定开放普通私信 API。联系方式引流也可能触发风控。解决方式：

- 先做轻回复和微信承接。
- 不使用抓包、逆向、非公开接口。
- 有官方授权后再做平台 Adapter。

### AI 风险

AI 可能编造价格、交期或承诺。解决方式：

- 价格和交期由规则控制。
- 高风险场景转人工。
- 所有回复经过安全护栏。

### 业务风险

设计、报价、生产和售后都涉及真实交付。解决方式：

- AI 只做辅助。
- 最终报价、改图确认、生产承诺由人工负责。

### 数据风险

聊天记录、客户联系方式、订单信息属于敏感数据。解决方式：

- 本地存储优先。
- API 加鉴权。
- 日志脱敏。
- 定期备份。

## 9. 项目完成定义

项目不是“能自动回复”就算完成。

真正完成应满足：

- 客户从平台来，能被记录。
- 客户加微信后，能被识别来源。
- 智能客服能自然回复常见问题。
- 能筛选高价值客户。
- 能提醒人工处理关键客户。
- 能补齐报价信息。
- 能提取出图提示词。
- 能进入出图任务。
- 能生成运营报表。
- 能从聊天记录沉淀知识库。
- 系统有测试、备份、日志、接口和控制台。

## 10. 后续 AI 助手执行约定

后续开发时，AI 助手应遵循：

- 先读本文档。
- 不猜平台接口。
- 不直接复制开源项目代码。
- 不破坏现有微信主链路。
- 每次改动都要跑测试或 smoke 检查。
- 新功能必须能在控制台或报表中被验证。
- 涉及客户隐私、平台接口、自动发送的功能，优先保守实现。

## 11. 当前最近优先级

优先级从高到低：

1. 平台线索录入和来源绑定。
2. 微信聊天记录导入与知识库学习。
3. 出图提示词提取和任务队列。
4. 控制台统一工作台继续完善。
5. API 鉴权和对外接口。
6. 官方平台接口调研和 Adapter 接入。

这条路线最适合当前业务：先把微信成交做深，再把抖音、小红书等平台作为获客入口接进来。
