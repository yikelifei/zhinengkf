# GitHub 开源智能客服项目调研

调研时间：2026-06-24

目标：参考 GitHub 上成熟项目的架构，完成当前礼盒定制智能客服系统。当前系统不建议整体换成某个开源项目，而是保留自己的业务核心，吸收成熟项目的模块设计。

## 结论

当前项目应该走“轻量业务定制系统”路线，不要直接改造成 Chatwoot、Dify 或 FastGPT 这种大平台。

推荐组合：

- 学 Chatwoot：统一收件箱、人工协作、标签、客户资料、报表。
- 学 Rasa：流程状态机、严格业务逻辑、人工接管规则。
- 学 Dify/FastGPT：知识库、RAG、工作流、工具调用。
- 学 CowAgent/chatgpt-on-wechat：多模型、多渠道、插件化。
- 学 Wechaty：渠道适配器思想，但谨慎使用个人微信非官方协议。

## 重点项目

| 项目 | GitHub | 适合借鉴 | 不建议照搬 |
| --- | --- | --- | --- |
| Chatwoot | https://github.com/chatwoot/chatwoot | 统一收件箱、客户资料、标签、人工协作、报表、自动分配 | 整体太重，国内平台接入仍要自己做 |
| Rasa | https://github.com/RasaHQ/rasa | 对话状态、流程、规则、Action、人工接管 | 传统 NLU 训练成本高，项目处于维护模式 |
| Dify | https://github.com/langgenius/dify | Agent 工作流、知识库、工具调用、模型管理 | 平台很重，许可证有附加条件 |
| FastGPT | https://github.com/labring/FastGPT | 知识库问答、RAG、可视化工作流、OpenAPI | 商用/SaaS 有许可证限制 |
| CowAgent | https://github.com/zhayujie/CowAgent | 多模型、多渠道、插件、技能、记忆 | 更偏通用 Agent，不是成交型客服 |

## 可以直接吸收的设计

### 1. 统一收件箱

Chatwoot 的核心价值不是 AI，而是把不同渠道的客户对话集中到一个 inbox。当前项目应该继续完善 Web 控制台：

- 全部客户消息
- 按平台筛选：微信、抖音、小红书
- 按状态筛选：待回复、已回复、需人工、已加微信、高价值
- 客户资料侧栏
- 对话标签
- 人工备注
- 跟进提醒
- 报表

当前项目已经有 `ChannelHub` 的基础，应继续把控制台做成真正的统一工作台。

### 2. 严格业务流程

Rasa 的核心经验是：LLM 可以理解用户，但关键业务步骤必须由代码控制。

当前礼盒定制业务不能让模型自由发挥报价和承诺交期，应该用状态机控制：

```text
新咨询
→ 收集需求
→ 判断是否高价值
→ 补齐报价字段
→ 生成初步方案
→ 人工报价/出图
→ 订单交接
→ 售后跟进
```

每一步必须有进入条件、退出条件和人工接管条件。

### 3. 知识库/RAG

Dify 和 FastGPT 的共同思路是：知识库不是简单模板，而是“资料处理 + 检索 + 引用 + 兜底”。

当前项目应把知识库升级成三层：

1. 固定规则：禁用承诺、价格边界、交期边界、敏感词。
2. FAQ：常见问题标准答案。
3. 案例知识：历史成交客户、礼盒方案、设计案例、客户异议。

微信聊天记录导入后，不应直接全部塞进模型，而要先做：

- 清洗无效聊天
- 提取真实客服高转化话术
- 提取常见问题
- 提取客户异议
- 人工审核后进入知识库

### 4. 多渠道适配器

CowAgent 和 Wechaty 都采用“渠道层”和“业务层”分离。当前项目已经开始这么做：

```text
平台 Adapter
→ ChannelHub
→ 智能客服大脑
→ 原平台回复 / 微信承接
```

后续新增平台时，只写 Adapter，不改客服大脑。

### 5. 人工接管

Chatwoot 和 Rasa 都强调人工协作。当前项目必须坚持：

- 投诉、退款、明确不满：人工接管
- 价格特别大、交期特别急：人工接管
- 客户要最终报价：人工接管
- 客户要改图：人工接管
- 模型信心低：人工接管

AI 负责筛选、收集、整理、建议；最终高风险动作由人工处理。

### 6. 线索评分

通用开源客服项目多数不懂你的业务，所以你自己的高价值客户评分是核心竞争力。

评分维度应继续强化：

- 数量
- 预算
- 交期
- 城市
- 企业采购
- 是否需要设计
- 是否要开票
- 是否愿意加微信
- 是否重复追问
- 来源平台

## 不建议直接照搬的内容

1. 不直接复制 AGPL 或附加许可证项目代码。

Chatwoot、Dify、FastGPT 都要认真看许可证。可以学架构，不要直接复制核心代码进商业项目。

2. 不依赖非官方私信接口。

抖音、小红书的私信和联系方式引流都有平台规则风险。当前更适合做“抖音/小红书获客，微信承接成交”。

3. 不把系统做成大而全平台。

当前项目最重要的是礼盒定制成交，不是做一个通用客服 SaaS。先做到自己业务闭环稳定，再考虑多租户和通用化。

## 推荐完成路线

### 第一阶段：微信成交主链路

- 微信智能回复稳定
- 知识库可维护
- 高价值客户筛选
- 报价字段补齐
- 人工接管
- 跟进报表
- 出图提示词提取

### 第二阶段：抖音/小红书获客承接

- 平台来源线索录入
- 加微信后的来源识别
- 平台引流话术库
- 线索与微信客户绑定
- 来源平台转化率统计

### 第三阶段：知识库自动学习

- 导入微信聊天记录
- 提取真实客服话术
- 提取 FAQ
- 提取异议处理
- 人工审核入库
- 定期生成优化建议

### 第四阶段：统一工作台

- 多平台线索列表
- 微信对话列表
- 高价值客户列表
- 人工接管队列
- 出图任务队列
- 今日待办
- 日报/周报

### 第五阶段：开放 API

给其他软件调用：

- `/api/chat/reply`
- `/api/leads/import`
- `/api/leads/high-value`
- `/api/image-prompts/extract`
- `/api/image-jobs`
- `/api/knowledge/search`

API 必须有鉴权、日志、限流和人工审核开关。

## 最终项目形态

推荐最终架构：

```text
抖音/小红书内容获客
        ↓
平台线索登记/半自动回复
        ↓
客户加微信
        ↓
微信智能客服
        ↓
知识库 + 业务规则 + 高价值评分
        ↓
报价准备 / 出图提示词 / 人工接管
        ↓
成交跟进 / 报表复盘 / 知识库迭代
```

这比“直接做全平台自动私信客服”更稳，也更适合商业化。

## 当前项目下一步建议

优先做这 5 个模块：

1. 平台线索录入：记录客户来自抖音还是小红书。
2. 微信来源绑定：客户加微信后绑定原平台来源。
3. 引流话术库：不同平台不同话术，避免违规硬引流。
4. 知识库学习器：从微信聊天记录提取高质量问答和话术。
5. 出图任务队列：从客户需求提取提示词，交给出图软件。
